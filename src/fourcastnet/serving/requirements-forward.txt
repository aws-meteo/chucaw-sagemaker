# Phase 2 (forward inference) serving requirements for FourCastNet.
#
# This file is the OPT-IN, heavier counterpart to requirements.txt. It is bundled into
# model.tar.gz only when packaging a forward-capable artifact:
#
#   python scripts/package_fourcastnet_model.py \
#     --requirements-file requirements-forward.txt --layout self-contained
#
# Keep the Phase 1 (metadata_only canary) build on the minimal requirements.txt so the
# cheap canary container starts fast and is not blocked by a heavy backend install.
#
# IMPORTANT — backbone.ckpt provenance (resolves the Phase-2 backend question):
# The shipped assets (backbone.ckpt + global_means.npy + global_stds.npy) are the
# NVlabs/FourCastNet pretrained AFNO backbone (Pathak et al. 2022), BSD-3-Clause.
# The checkpoint is NVlabs-format, NOT Modulus/PhysicsNeMo-format:
#   - torch.load(...) -> dict with key 'model_state' (not 'model_state_dict')
#   - weight keys are prefixed 'module.' (strip the first 7 chars)
#   - architecture is AFNONet from NVlabs networks/afnonet.py, built with params:
#       img_size=(720,1440), patch_size=(8,8), in_chans=20, out_chans=20,
#       embed_dim=768, depth=12
# Therefore nvidia-physicsnemo's FourCastNet class CANNOT load this checkpoint directly,
# and inference.py _probe_backend (which looks for modulus/physicsnemo/fourcastnet
# classes + load_from_checkpoint) does NOT match it. To run real forward inference you
# must vendor the NVlabs AFNONet network code (networks/afnonet.py + utils/img_utils.py,
# BSD-3 notice retained) into the serving bundle and load 'model_state' with the
# 'module.' prefix stripped. The only extra pip deps AFNONet needs are timm + einops.
